# `snake.html` 审查报告

## 结论

发现 3 个值得优先处理的问题，其中 1 个会直接导致玩法错误，1 个会在接近满盘时触发死循环风险，1 个会让暂停态继续接收方向输入。另有少量可维护性问题，但优先级低于前述行为问题。

本次结论基于静态代码审查，未做浏览器中的完整交互回归验证。

## Findings

### 1. 高：撞到“本回合即将移开的尾巴”也会被判定为死亡

- 位置：`update()`，约 `snake.html:225-253`
- 问题：当前先用 `snake.some(...)` 检查整条蛇，再决定是否 `pop()` 尾巴。这样会把“头移动到当前尾巴所在格，但该尾巴本回合本应移开”的合法移动误判成自撞。
- 影响：这是标准贪吃蛇里的真实玩法错误，会出现玩家明明走法合法却直接死亡的情况。
- 建议：碰撞检测时应区分“本回合是否吃到食物”。如果没有吃到食物，应在自撞判断里排除尾节点，或先计算出下一帧的有效身体集合后再判断。

### 2. 中：食物生成在棋盘接近填满时可能进入无限循环，且没有胜利态

- 位置：`spawnFood()`，约 `snake.html:208-220`；补食逻辑在 `snake.html:249`
- 问题：`spawnFood()` 用 `do...while` 一直随机，默认假设一定存在空位；但当蛇身和已有食物几乎占满棋盘，尤其是空位少于 `MIN_FOOD` 所需数量时，这个循环可能永远找不到可用格子。
- 影响：极端情况下游戏会卡死，尤其是分数很高、接近通关时最容易暴露。
- 建议：生成前先统计空格数量；若没有空格或空格不足，应进入胜利/通关状态，或只补到剩余可用数量为止，而不是无条件补满到 `MIN_FOOD`。

### 3. 低：暂停状态下仍然接受方向输入，恢复后会立即转向

- 位置：键盘处理逻辑，约 `snake.html:625-682`
- 问题：`paused === true` 时，方向键和 `WASD` 仍会更新 `nextDir`。虽然游戏画面暂停了，但输入状态还在变化。
- 影响：玩家恢复游戏后，蛇可能立刻按暂停期间缓存的方向转向，行为和“暂停冻结一切输入”的常见预期不一致。
- 建议：在暂停态屏蔽方向输入，或明确把这种行为设计成“允许预输入”，并在界面文案里说明。

## 其他观察

- `sortBy()` 当前未被使用，`dizzy` 状态值也基本没有参与判定逻辑，属于可清理的冗余代码。
- `highScore` 只保存在内存里，刷新页面会清零。这更像产品取舍，不算当前实现里的硬 bug。

## 建议修复顺序

1. 先修复尾巴误判自撞问题。
2. 再处理满盘时的补食与胜利态逻辑。
3. 最后统一暂停态的输入策略，并顺手清理冗余状态。

## 测试缺口

- 需要手动验证“头进入尾巴原位置但尾巴会同步移开”的场景。
- 需要验证蛇身接近占满棋盘时不会卡死。
- 需要验证暂停前、暂停中、恢复后的方向输入行为是否符合预期。
